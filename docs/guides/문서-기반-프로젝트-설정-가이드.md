# 📄 문서 기반 프로젝트 설정 가이드

> 템플릿 문서를 활용한 프로젝트 초기 설정 및 재사용 방법론

---

## 🎯 개요

이 문서는 **vive-md** 프로젝트에서 사용하는 **문서 기반 프로젝트 설정 방식**에 대해 자세히 설명합니다. 이 방식의 핵심은 다음과 같습니다:

1. **템플릿 문서 보관소**: 모든 개발 가이드와 산출물 템플릿을 중앙 집중식으로 관리
2. **복사-붙여넣기 재사용**: 필요한 문서만 선택적으로 복사하여 새 프로젝트에 적용
3. **AI 기반 코드 생성**: 복사된 문서를 Claude Code의 컨텍스트로 활용하여 일관된 코드 생성

## 📦 템플릿 보관소 구조

### 1. 기술 스택 가이드 (`templates/{stack}/`)

각 기술 스택별 상세 개발 가이드를 포함합니다.

| 디렉토리 | 파일 | 설명 | 분량 |
|----------|------|------|------|
| `spring-boot/` | `Spring-Boot-개발가이드.md` | Spring Boot 3.x, Java 17+, Gradle | ~3,200줄 |
| `vue/` | `Vue3-개발가이드.md` | Vue 3, Composition API, TypeScript, Vite | ~3,000줄 |
| `react/` | `React-개발가이드.md` | React 18+, TypeScript, Vite | ~3,000줄 |
| `nextjs/` | `NextJS-개발가이드.md` | Next.js 14+, App Router, TypeScript | ~2,700줄 |

#### 가이드 공통 구조

모든 개발 가이드는 다음 6개 섹션으로 구성됩니다:

```markdown
1. 프로젝트 개요
   - 기술 스택 버전
   - 아키텍처 원칙
   - 프로젝트 구조

2. 코딩 컨벤션
   - 명명 규칙
   - 코드 스타일
   - 파일/디렉토리 구조

3. 보안 필수사항
   - 보안 체크리스트
   - 취약점 대응
   - 인증/인가 패턴

4. 생산성 가이드
   - 개발 워크플로우
   - 도구 설정
   - 디버깅 팁

5. 성능 최적화
   - 성능 패턴
   - 모범 사례
   - 측정 방법

6. Gotchas
   - 흔한 실수
   - 해결책
   - 주의사항
```

### 2. 워터폴 산출물 템플릿 (`templates/waterfall/`)

전통적인 워터폴 방법론의 각 단계별 산출물 템플릿을 제공합니다.

```

├── 01-요구사항분석/

│
├── 02-시스템설계/
│   ├── 시스템아키텍처설계서-SAD.md     # 아키텍처 개요, 컴포넌트
│   ├── 데이터베이스설계서.md          # ERD, 테이블 정의
│   ├── API설계서.md                   # 엔드포인트, 요청/응답
│   └── 화면설계서.md                  # UI 구성, 네비게이션
│
├── 03-상세설계/
│   └── 상세설계서.md                  # 클래스, 시퀀스, 알고리즘
│
├── 05-테스트/
│   ├── 테스트계획서.md                # 전략, 범위, 일정
│   ├── 테스트케이스.md                # 시나리오, 입력, 예상 결과
│   └── 테스트결과보고서.md            # 실행 결과, 결함
│
├── 06-배포/
│   ├── 배포계획서.md                  # 절차, 환경, 롤백
│   └── 운영가이드.md                  # 모니터링, 장애 대응
│
├── 07-유지보수/
│   └── 유지보수계획서.md              # 변경 관리, 개선 계획

#### 템플릿 특징

- **`[placeholder]`** 표시: 프로젝트별 내용을 채울 위치 명시
- **Mermaid 다이어그램**: flowchart, sequenceDiagram, erDiagram, classDiagram, gantt 포함
- **표준 형식**: SI 프로젝트에서 요구하는 표준 문서 형식 준수

### 3. 크로스커팅 가이드 (`templates/{cross-cutting}/`)

모든 기술 스택에 공통으로 적용되는 가이드입니다.

## 🔄 프로젝트 설정 워크플로우

### 단계 1: 템플릿 선택

프로젝트 요구사항에 맞는 템플릿을 선택합니다.

```

│                                                            │
│  🌐 풀스택 웹 애플리케이션                                    │

│     ├── 프론트엔드: React-개발가이드.md 또는 Vue3-개발가이드.md │
│     └── 공통: 보안-가이드.md                                 │

### 단계 2: 문서 복사

선택한 템플릿을 새 프로젝트로 복사합니다.

#### 시나리오 A: 단일 프로젝트 (Spring Boot)

```bash
# 1. 프로젝트 디렉토리 생성
mkdir my-spring-project && cd my-spring-project

# 2. 개발 가이드 복사 → CLAUDE.md로 이름 변경
cp /path/to/vive-md/templates/spring-boot/Spring-Boot-개발가이드.md CLAUDE.md

# 3. (선택) 보안 가이드 추가
cp /path/to/vive-md/templates/security/보안-가이드.md 보안-가이드.md

# 4. (선택) 워터폴 산출물 템플릿 복사
mkdir -p docs
cp -r /path/to/vive-md/templates/waterfall/* docs/
```

#### 시나리오 B: 풀스택 프로젝트 (Spring Boot + React)

```bash
# 1. 프로젝트 루트 생성
mkdir my-fullstack-project && cd my-fullstack-project

# 2. 백엔드 설정
mkdir backend
cp /path/to/vive-md/templates/spring-boot/Spring-Boot-개발가이드.md backend/CLAUDE.md

# 3. 프론트엔드 설정
mkdir frontend
cp /path/to/vive-md/templates/react/React-개발가이드.md frontend/CLAUDE.md

# 4. 공통 문서
mkdir docs
cp /path/to/vive-md/templates/security/보안-가이드.md docs/
cp -r /path/to/vive-md/templates/waterfall/* docs/
```

#### 시나리오 C: SI 프로젝트 (워터폴 중심)

```bash
# 1. 프로젝트 디렉토리 생성
mkdir si-project && cd si-project

# 2. 워터폴 산출물 전체 복사
mkdir -p docs
cp -r /path/to/vive-md/templates/waterfall/* docs/

# 3. 개발 가이드 추가 (해당 기술 스택)
cp /path/to/vive-md/templates/spring-boot/Spring-Boot-개발가이드.md docs/개발가이드.md

# 4. 보안 가이드 추가
cp /path/to/vive-md/templates/security/보안-가이드.md docs/
```

### 단계 3: Claude Code 컨텍스트 설정

복사된 `CLAUDE.md` 파일이 Claude Code의 프로젝트 컨텍스트로 자동 로드됩니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Claude Code 동작 원리                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 사용자가 프로젝트 디렉토리에서 Claude Code 실행               │
│                                                                  │
│  2. Claude Code가 자동으로 CLAUDE.md 탐색                        │
│     └── 현재 디렉토리 → 상위 디렉토리 순으로 검색                  │
│                                                                  │
│  3. CLAUDE.md 내용을 시스템 프롬프트에 포함                       │
│     └── 기술 스택, 코딩 컨벤션, 보안 규칙 등 로드                  │
│                                                                  │
│  4. 사용자 요청 처리                                             │
│     사용자: "로그인 기능 만들어줘"                                │
│     Claude: CLAUDE.md의 인증/인가 패턴을 기반으로 코드 생성        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2. 프로젝트별 커스터마이징

복사된 CLAUDE.md를 프로젝트 특성에 맞게 수정합니다.

```markdown
# CLAUDE.md 예시 (커스터마이징)

## 프로젝트 특화 규칙

### 이 프로젝트만의 규칙
- 모든 API 응답은 ApiResponse<T> 래퍼 사용
- 날짜 형식: ISO-8601 (YYYY-MM-DDTHH:mm:ssZ)
- 페이지네이션 기본값: size=20, page=0

### 팀 컨벤션
- 브랜치 명명: feature/TICKET-123-short-desc
- 커밋 메시지: [TICKET-123] 작업 내용
- PR 템플릿: .github/pull_request_template.md 사용
```

### 3. 버전 관리

템플릿 보관소를 Git 서브모듈로 관리할 수 있습니다.

```bash
# 1. 템플릿 보관소를 서브모듈로 추가
git submodule add https://github.com/your-org/vive-md.git templates/vive-md

# 2. 필요할 때마다 복사
cp templates/vive-md/templates/react/React-개발가이드.md CLAUDE.md

# 3. 템플릿 업데이트
git submodule update --remote templates/vive-md
```

### 4. 팀 공유용 템플릿 패키지

팀 전용 템플릿 패키지를 만들어 공유합니다.

```bash
# team-templates/ 디렉토리 구조
team-templates/
├── backend/
│   ├── CLAUDE.md              # Spring Boot + 팀 규칙
│   ├── docker-compose.yml     # 팀 표준 인프라
│   └── application-dev.yml    # 팀 표준 설정
├── frontend/
│   ├── CLAUDE.md              # React + 팀 규칙
│   └── .eslintrc.js           # 팀 표준 린트 설정
└── scripts/
    └── init-project.sh        # 프로젝트 초기화 스크립트

# 초기화 스크립트 예시 (init-project.sh)
#!/bin/bash
PROJECT_NAME=$1
STACK=$2  # backend, frontend, fullstack

mkdir $PROJECT_NAME
cp -r team-templates/$STACK/* $PROJECT_NAME/
echo "프로젝트 $PROJECT_NAME 초기화 완료!"
```

## 📊 효과 비교

### Before: 문서 기반 설정 미사용

| 항목 | 문제점 |
|------|--------|
| **프로젝트 초기화** | 매번 아키텍처 결정, 2~3일 소요 |
| **코드 일관성** | 개발자마다 다른 스타일, 코드 리뷰 부담 |
| **보안** | 누락되는 보안 체크리스트, 반복적인 취약점 |
| **온보딩** | 신규 개발자가 컨벤션 익히는 데 1주일+ |
| **산출물 작성** | 표준 없이 각자 작성, 품질 불일치 |

### After: 문서 기반 설정 사용

| 항목 | 개선 효과 |
|------|----------|
| **프로젝트 초기화** | `cp CLAUDE.md`로 즉시 완료, 10분 소요 |
| **코드 일관성** | AI가 가이드 기반 코드 생성, 자동 일관성 유지 |
| **보안** | 체크리스트 기반 자동 검증, 취약점 사전 방지 |
| **온보딩** | CLAUDE.md 읽으면 즉시 이해, 1시간 |
| **산출물 작성** | 템플릿 기반 자동 생성, 표준 품질 보장 |
