## 1. 유지보수 유형 정의

### 1.1 유지보수 유형 분류

#### 수정 유지보수 (Corrective Maintenance)

| 항목 | 내용 |
|------|------|
| **정의** | 운영 중 발견된 결함(버그)을 수정하는 활동 |
| **목적** | 시스템의 정상 동작을 복구하고 사용자에게 미치는 영향을 최소화 |
| **트리거** | 사용자 보고, 모니터링 알림, 내부 발견 |
| **우선순위** | 결함 심각도에 따라 즉시 ~ 차기 릴리스 |

**예시**:
- 특정 조건에서 데이터가 잘못 저장되는 버그 수정
- 에러 핸들링 누락으로 인한 500 에러 수정
- 동시성 이슈로 인한 데이터 정합성 오류 수정
- 브라우저 업데이트로 발생한 호환성 문제 수정

#### 적응 유지보수 (Adaptive Maintenance)

| 항목 | 내용 |
|------|------|
| **정의** | 변화하는 환경(OS, 브라우저, 라이브러리, 규정 등)에 시스템을 적응시키는 활동 |
| **목적** | 변경된 환경에서도 시스템이 정상적으로 동작하도록 보장 |
| **트리거** | 외부 환경 변화 (OS 업데이트, API 변경, 규정 변경 등) |
| **우선순위** | 환경 변화 시점 및 영향도에 따라 결정 |

**예시**:
- 새 브라우저 버전 호환성 대응
- 외부 API 버전 업그레이드 대응
- 의존성 패키지 보안 패치 적용
- 개인정보보호법 변경에 따른 기능 수정
- 클라우드 서비스 정책 변경 대응

#### 완전 유지보수 (Perfective Maintenance)

| 항목 | 내용 |
|------|------|
| **정의** | 사용자 요구사항 변경이나 기능 개선을 통해 시스템을 향상시키는 활동 |
| **목적** | 사용자 만족도 향상 및 비즈니스 가치 증대 |
| **트리거** | 사용자 피드백, 비즈니스 요구, 경쟁 대응 |
| **우선순위** | 비즈니스 가치 및 사용자 영향도에 따라 결정 |

**예시**:
- 사용자 피드백 기반 UX 개선
- 성능 최적화 (응답 시간 개선, 쿼리 최적화)
- 새로운 기능 추가
- 기존 기능 개선 (검색 정확도 향상, UI 리디자인)
- 접근성 개선

#### 예방 유지보수 (Preventive Maintenance)

| 항목 | 내용 |
|------|------|
| **정의** | 잠재적 문제를 사전에 식별하고 예방하기 위한 활동 |
| **목적** | 시스템 안정성 확보 및 장애 예방 |
| **트리거** | 정기 점검, 기술 부채 검토, 보안 감사 |
| **우선순위** | 위험도 및 영향도에 따라 결정 |

**예시**:
- 코드 리팩토링 (복잡도 감소, 가독성 향상)
- 의존성 패키지 정기 업데이트
- 데이터베이스 인덱스 최적화
- 보안 취약점 사전 점검 및 패치
- 테스트 코드 보강
- 기술 부채 해소

## 2. 변경 요청(CR) 프로세스

### 2.1 CR 프로세스 흐름도

```mermaid
flowchart TD
    A[변경 요청 접수] --> B[CR 등록 및 분류]
    B --> C[영향 분석]
    C --> D{영향도 / 규모 판단}

    D -->|소규모 변경| E1[팀 리드 승인]
    D -->|중규모 변경| E2[변경 관리 위원회 (CCB) 검토]
    D -->|대규모 변경| E3[경영진 승인]

    E1 --> F{승인?}
    E2 --> F
    E3 --> F

    F -->|승인| G[구현 계획 수립]
    F -->|반려| H[요청자 피드백 전달]
    F -->|보류| I[보류 사유 기록 및 재검토 일정 설정]

    H --> J[종료 또는 재요청]
    I --> B

    G --> K[구현]
    K --> L[테스트]
    L --> M{테스트 통과?}
    M -->|예| N[배포]
    M -->|아니오| K

    N --> O[검증 및 모니터링]
    O --> P[변경 완료 및 종료]
```

| **Minor (P3)** | 부가 기능 장애, 경미한 성능 저하 | [2]시간 | [4]시간 | [3]영업일 | [10]영업일 |

## 4. 기술 부채 관리

### 4.1 기술 부채 분류

| 분류 | 설명 | 예시 | 영향 |
|------|------|------|------|
| **코드 부채** | 코드 품질 저하, 복잡도 증가, 중복 코드 | 하드코딩된 값, 매직 넘버, 긴 함수, 복잡한 조건문 | 유지보수 비용 증가, 버그 발생률 증가 |
| **아키텍처 부채** | 부적절한 설계, 확장성 제한 | 모놀리식 구조의 한계, 부적절한 의존성, 결합도 높음 | 확장 어려움, 성능 병목 |
| **테스트 부채** | 테스트 부족, 테스트 품질 저하 | 낮은 코드 커버리지, E2E 테스트 부재, 깨진 테스트 방치 | 회귀 버그 위험, 리팩토링 어려움 |
| **문서 부채** | 문서 부재 또는 오래된 문서 | API 문서 미갱신, 운영 매뉴얼 부재, 주석 미갱신 | 온보딩 어려움, 운영 실수 |
| **인프라 부채** | 인프라 설정 관리 부재, 수동 운영 | IaC 미적용, 수동 배포, 모니터링 부족 | 운영 실수, 복구 지연, 확장 어려움 |
