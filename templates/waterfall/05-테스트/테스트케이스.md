## 1. 테스트 케이스 관리 규칙

### 1.1 작성 기준

| 항목 | 규칙 |
|------|------|
| **하나의 TC = 하나의 검증 목적** | 각 테스트 케이스는 단일 기능/조건을 검증 |
| **독립 실행 가능** | 다른 TC에 의존하지 않고 단독 실행 가능해야 함 |
| **재현 가능** | 동일 조건에서 반복 실행 시 동일 결과 |
| **사전조건 명시** | 테스트 수행에 필요한 전제 조건을 명확히 기술 |
| **기대결과 명확** | 통과/실패 판단 기준이 객관적이고 측정 가능 |
| **요구사항 추적** | 관련 요구사항 ID를 반드시 기재 |

### 1.2 네이밍 규칙

| 접두사 | 유형 | 형식 | 예시 |
|--------|------|------|------|
| TC-FN | 기능 테스트 | TC-FN-[모듈]-[번호] | TC-FN-AUTH-001 |
| TC-PF | 성능 테스트 | TC-PF-[대상]-[번호] | TC-PF-API-001 |
| TC-SC | 보안 테스트 | TC-SC-[유형]-[번호] | TC-SC-XSS-001 |
| TC-CP | 호환성 테스트 | TC-CP-[환경]-[번호] | TC-CP-BRW-001 |
| TC-AC | 접근성 테스트 | TC-AC-[기준]-[번호] | TC-AC-KBD-001 |
| TC-RG | 회귀 테스트 | TC-RG-[모듈]-[번호] | TC-RG-AUTH-001 |

### 1.3 상태 관리

| 상태 | 설명 | 표기 |
|------|------|------|
| Not Executed | 미실행 | - |
| Pass | 기대결과와 일치 | PASS |
| Fail | 기대결과와 불일치 (결함 등록) | FAIL |
| Blocked | 사전조건 미충족 등으로 실행 불가 | BLOCKED |
| Skipped | 의도적 건너뜀 (사유 기재) | SKIPPED |
| N/A | 해당 없음 (환경/조건 변경) | N/A |
